## 1. Назначение документа
Документ содержит функциональные и нефункциональные требования к автоматизированной системе контроля параметров водной среды на основе интернета вещей (IoT). Система предназначена для использования на предприятиях аквакультуры (рыбоводства, выращивания ракообразных/моллюсков).

## 2. Функциональные требования

Данный раздел описывает, что именно должна делать система.

### 2.1. Сбор данных (Периферийный уровень)
*   **F-01: Мониторинг физических параметров:** Система должна обеспечивать непрерывное измерение следующих параметров водной среды в реальном времени:
    *   Температура воды.
    *   Уровень pH (кислотность).
    *   Концентрация растворенного кислорода (DO).
    *   Уровень электропроводности / общей минерализации (TDS).
    *   Уровень мутности / прозрачности (опционально).
*   **F-02: Мониторинг химических параметров:** Система должна поддерживать подключение датчиков для контроля содержания в воде:
    *   Аммония/аммиака (NH4/NH3).
    *   Нитритов (NO2-) и нитратов (NO3-).
*   **F-03: Опрос датчиков:** Микроконтроллеры (шлюзы) должны опрашивать подключенные датчики с заданной периодичностью (не реже 1 раза в 5-10 минут для динамических параметров, таких как кислород и температура).
*   **F-04: Идентификация точки замера:** Каждый пакет данных должен быть привязан к конкретному бассейну/участку (идентификатор узла IoT).

### 2.2. Передача и хранение данных (Сетевой уровень)
*   **F-05: Передача данных:** Обеспечить надежную передачу данных с датчиков на центральный сервер (on-premise или облачный) по беспроводным каналам связи (Wi-Fi, LoRaWAN, сотовые сети 4G в зависимости от условий предприятия).
*   **F-06: Хранение данных:** Система должна накапливать и хранить исторические данные измерений в структурированной базе данных (БД) для последующего анализа (Big Data).
*   **F-07: Сохранность при обрывах связи:** Локальные контроллеры должны иметь буфер памяти для хранения данных при потере соединения с сервером и автоматическую отправку накопленных данных после восстановления канала связи.

### 2.3. Обработка, визуализация и оповещение
*   **F-08: Индикация на пульте управления:** Разработка графического интерфейса пользователя (HMI/SCADA) для отображения текущих параметров по каждому бассейну/узлу в реальном времени.
*   **F-09: Настройка пороговых значений:** Предоставить пользователю (оператору) возможность настраивать верхние и нижние критические границы (пределы) для каждого контролируемого параметра.
*   **F-10: Оперативное уведомление об отклонениях:**
    *   При выходе значения параметра за установленные пределы система должна генерировать событие.
    *   Отправка уведомлений персоналу (дежурному оператору, технологу) по следующим каналам: звуковой сигнал на пульте, всплывающее окно в интерфейсе, SMS/E-mail или Push-уведомления в мессенджер (Telegram) ответственным лицам.
*   **F-11: Визуализация трендов:** Возможность построения графиков изменения параметров за произвольный промежуток времени (час, день, неделя, месяц).

### 2.4. Аналитика и управление
*   **F-12: Формирование отчетов:** Автоматическая генерация отчетов (сменных, суточных) о параметрах среды.
*   **F-13: Накопление массива данных для корреляционного анализа:** Система должна обеспечивать выгрузку накопленных данных во внешние аналитические пакеты (например, CSV, Excel) или иметь встроенные инструменты для поиска корреляций между параметрами водной среды (например, влияние роста температуры на уровень pH) и другими входными данными производства (внесение кормов, замена воды и т.д.).

## 3. Нефункциональные требования

Данный раздел описывает, какой должна быть система по качеству, надежности и безопасности.

### 3.1. Требования к надежности
*   **NF-01: Отказоустойчивость:** Система должна сохранять работоспособность при отказе одного из датчиков (остальные узлы продолжают сбор данных).
*   **NF-02: Бесперебойное питание:** Узлы сбора данных (шлюзы) должны быть подключены через источники бесперебойного питания (ИБП) или иметь резервное питание для работы при отключении электроэнергии не менее 4 часов.
*   **NF-03: Восстановление после сбоев:** Время восстановления работоспособности программного обеспечения после сбоя не должно превышать 5 минут (автоматический перезапуск служб).

### 3.2. Требования к точности и производительности
*   **NF-04: Точность измерений:** Класс точности используемых датчиков должен соответствовать технологическим нормам производства (например, для растворенного кислорода погрешность не более 0.1 мг/л, для pH — 0.05).
*   **NF-05: Период обновления данных:** Максимальная задержка от момента физического измерения до отображения данных на пульте управления не должна превышать 2 минуты.
*   **NF-06: Масштабируемость:** Обеспечение возможности подключения не менее 50 новых измерительных узлов без необходимости изменения архитектуры серверного ПО.

### 3.3. Требования к безопасности
*   **NF-07: Защита данных:** Передача данных от датчиков к серверу должна осуществляться по зашифрованным каналам (TLS/SSL).
*   **NF-08: Аутентификация и авторизация:** Доступ к пульту управления и истории данных должен быть защищен паролем. Разграничение прав доступа (администратор, оператор, технолог).
*   **NF-09: Защита от ложных срабатываний:** Алгоритмы оповещения должны содержать логику подтверждения тревоги (например, фильтрация резких скачков датчика для исключения ложных уведомлений).

### 3.4. Требования к эксплуатации
*   **NF-10: Климатическое исполнение:** Уличные блоки и датчики должны иметь степень защиты не ниже IP65 (пылевлагозащита) для работы в условиях повышенной влажности.
*   **NF-11: Метрологическое обслуживание:** Конструкция датчиков должна предусматривать возможность их демонтажа для плановой калибровки и очистки.
*   **NF-12: Интерфейс пользователя:** Интерфейс пульта управления должен быть интуитивно понятным, не требующим специального программирования от оператора. Аварийные сообщения должны выделяться красным цветом и миганием.
